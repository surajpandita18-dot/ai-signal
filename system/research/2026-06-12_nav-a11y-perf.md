## 0. Headline findings (read this even if you read nothing else)

1. **Archivo Expanded is silently broken in production.** The Google Fonts CSS
   request at `https://fonts.googleapis.com/css2?family=Archivo+Expanded:...`
   returns **HTTP 400 "Font family not found"** — the family is not in Google
   Fonts under that name. Every section relying on it (`.hstep .hn`,
   `.spotlight .stat`, `.interview .iv-q .q`, `.reality h3`, `.sig h4`,
   `.poll .q`, `.signal li i`) is rendering in `system-ui, sans-serif`. The
   loaded `FontFace` list confirms zero Archivo Expanded entries
   (`archivoExpandedEntries: []`).
2. **Five interactive UI primitives are mouse-only.** All five lens track chips,
   three poll options, and the `.copyable` code block on the issue page are
   `<span>`/`<div>` with `onClick` and no `tabindex`/`role` — they don't appear
   in the SR tree, can't be tabbed to, can't be activated with Space/Enter.
3. **The site has zero primary navigation.** The landing page has **zero**
   anchor tags. The archive page has one link (back to home). The 404 has
   none. A first-time reader on the landing has no way to find the about page,
   the archive, or issue 001 except by guessing URLs or filling the form.

## 2. Information architecture

The mental model the site asks a first-time visitor to build is:
> "There is a newsletter. The form is the only door. There is nothing else."

That's deliberate restraint (per `CLAUDE.md` rule 2) and arguably right for the
landing's conversion job. But every page repeats the pattern — even the
issue page, which is where curious-but-not-yet-converted readers will land
from social shares. The result is that:

- The site has **no shared chrome**. Each page builds its own masthead (one
  static on landing/about, one data-driven on issues). Mast usage is
  inconsistent: landing/about wordmark is a `<span>`; about wordmark is an
  `<a href="/">`. Same component reads as both heading and link in different
  contexts.
- There are **no contextual landmarks** beyond `<main>`. Zero `<nav>`,
  `<footer>`, or skip-link on any page.
- The issue page has **two `<h2>` elements** in a ~5000px article. The other
  eight "Section" components (Job Signal, Toolbox, Reality Check, India
  Signal, Sponsor, etc.) render their titles as `<span class="nm-lab">`, not
  `<h2>`. Sighted users see them as headings; SR users do not.

**Recommendation:** Keep restraint, but add a tiny dual-purpose footer chrome:
one row, low-contrast, on every page, with three links: `/`, `/about`,
`/archive`. That alone closes 12 of the 14 missing nav edges.

### 5.1 Font loading

8 woff2 files on the issue page, all from `_next/static/media/` (self-hosted
via `next/font`), all returning 200, all between 18–65 KB. Total font
weight: ~327 KB transferred (already counted above).

`Archivo Expanded` is loaded via a Google Fonts CSS link in `layout.tsx:96`,
**which returns 400.** The fallback `system-ui, sans-serif` is what
actually renders. This is the highest-impact bug in the whole audit — every
"display weight" treatment in the issue is broken.

Two paths forward:
1. **Drop the dependency.** Replace `'Archivo Expanded'` with the `Archivo`
   variable already loaded by `next/font`, at weight 800, plus
   `font-stretch: 125%` (Archivo supports a width axis). Visually closer than
   the current `system-ui` fallback.
2. **Self-host a real Archivo Expanded font.** Add to `next/font/local` with
   the actual `.woff2` files. Heavier change, more fidelity.

### 5.2 Other perf notes

- The preconnects for `fonts.googleapis.com` / `fonts.gstatic.com` in
  `layout.tsx:89-94` are now wasted — only the broken Archivo Expanded link
  ever used them. Remove them when the font fix lands.
- Archive LCP at 2572ms is the only page touching Supabase server-side
  (`createServerSupabaseClient` + 1 query). Currently 1 row. With ~50 rows the
  payload is still trivial; the latency is the Supabase round-trip itself.
  Cache the SELECT for 60s with `revalidate`.

## 8. Top 10 fixes ranked by impact

### #1 — Fix Archivo Expanded (silently broken in prod)

- **File:** `src/app/layout.tsx:96`
- **Severity:** Critical (visual regression on every display-weight headline)
- **Effort:** S

Before:
```tsx
<link
  href="https://fonts.googleapis.com/css2?family=Archivo+Expanded:wght@600;700;800&display=swap"
  rel="stylesheet"
/>
<style>{`:root{--font-expanded:'Archivo Expanded',system-ui,sans-serif}`}</style>
```

After (path 1 — re-use the already-loaded Archivo variable):
```tsx
// Drop the <link> and the <style>. In tokens.css, alias:
// :root { --font-expanded: 'Archivo', system-ui, sans-serif; }
// Then in issue.css, anywhere using 'Archivo Expanded' add font-stretch: 125%.
```

Why: the Google Fonts CSS endpoint returns 400. Confirmed via `curl` and via
`document.fonts` enumeration in the live page. Every "Archivo Expanded"
selector currently renders `system-ui`. Either bind the variable to a font
that actually loads, or self-host the real `Archivo Expanded` files via
`next/font/local`.

### #2 — Make poll / lens-chip / copyable keyboard-accessible

- **Files:**
  - `src/components/interactive/PollClient.tsx:43-50`
  - `src/components/interactive/LensTrackPicker.tsx:45-58`
  - `src/components/interactive/CopyOnClick.tsx:42-46`
- **Severity:** Critical (a11y violation: WCAG 2.1.1 Keyboard)
- **Effort:** S

Before (`PollClient.tsx`):
```tsx
<span
  key={i}
  className={`opt${picked === o ? ' picked' : ''}`}
  onClick={() => vote(o)}
>
  {o}
</span>
```

After:
```tsx
<button
  key={i}
  type="button"
  className={`opt${picked === o ? ' picked' : ''}`}
  onClick={() => vote(o)}
  disabled={!!picked}
>
  {o}
</button>
```

(Same pattern for the chips in `LensTrackPicker.tsx` and the wrapper in
`CopyOnClick.tsx`.) `.poll .opt` and `.track-chip` CSS already styles `border`
and `padding` — buttons inherit `cursor: pointer` and won't visually change.

Why: today, these are mouse-only. A screen-reader user cannot vote in the
poll, cannot filter lenses, cannot copy the example code. WCAG 2.1.1.
