## Where we score worst (root causes)

The system-wide weakest signals, ordered:

1. **Email contrast (1/5 across 3 emails)** — the IssueEmail template uses
   `#7d776c` GREY (rgb 125,119,108) for eyebrow labels, source citations, and
   the "Why you care" italic line, on a cream `#F4F1E8` background. That ratio
   sits at ~3.6:1 — failing 4.5:1 body text. Either bump these to `#5a574e`
   (≥4.5:1) or upcast them to `font-size: 18px` so the large-text rule applies.
   *Highest-weight finding* (weight × 3 on contrast).

2. **Email palette (1/5)** — the IssueEmail uses ~10 distinct colour stops:
   INK, ACCENT, CLAY, FAINT, HAIR, GREY, DARK_BAND, plus the white-on-dark band
   inverse, plus the source-grey `#555`, plus the `#cfc7b8` referral subtitle.
   That's intentional editorial range, but the rubric is the rubric. If we want
   ≤6 hues we'd need to retire `#555` and `#cfc7b8` in favour of GREY +
   DARK_BAND tones.

3. **Line length (1–2/5 across 5 of 6 targets)** — measured against the 45–75 ch
   long-form ideal. At 600 px email columns with 15 px serif body, we run at
   roughly 35–45 ch — short. This may be a rubric-vs-reality mismatch (email
   columns at 600 px will never hit 75 ch with 15 px body). Decide:
   widen the column, drop the font size, or tune the rubric. Worth a product
   conversation, not a code change.

4. **Subject + preview length (email:002, email:003)** — these issues have hero
   headlines and TLDR-first-rows that exceed Litmus mobile previews. Editor
   intervention: tighten subjects to ≤50 chars and TLDR row 0 to ≤80 chars.

5. **Tap targets @390 (3/5 web landing, 1/5 /i/001)** — inline `<a>` links in
   paragraphs don't hit 44 px. Real but minor — most are body text links, not
   primary CTAs. The primary subscribe CTA tested ≥44 px.

## What's blocking

Strictly under the rubric's ship gate: **email:002 and email:003 are at 3.88
and 3.90**, both below the 4.0 SHIP threshold. The blocker for both is the
same:

- contrast on the small grey eyebrow / source labels (weight 3 × score 1)
- palette breadth (weight 1 × score 1)
- and for `email:003`, the preview text is over 90 chars after react-email
  padding is stripped.

Three pragmatic fixes that would clear both:

1. Bump GREY label colour from `#7d776c` to `#5a574e` (or upcast eyebrows
   to 18 px) — moves contrast from 1 → 5 on weight 3. Single change. Would
   take email:001 from 4.0 → ~4.4, email:002/003 from 3.9 → 4.3.
2. Trim the preview text in `content/issues/003.json` first TLDR row to
   ≤80 chars — moves preview from 2 → 5 on weight 1.
3. Subject-line tightening in `002.json` + `003.json` — soft win.

Decision needed from Suraj on the **line-length rule** before we rewrite for
that signal — the 45–75 ch ideal is correct for long-form prose, but tight
email columns may legitimately run shorter.

## Reproduction

```sh
node scripts/rubric.mjs all
# exit 0 → all targets ≥4.0
# exit 1 → at least one target in 3.0–3.9
# exit 2 → at least one target <3.0
```

Re-running this command on the same content (no code or content changes
in between) must produce identical aggregate numbers within rounding.
The runner enforces determinism by:

- Pinning Playwright viewports to fixed integer pixel sizes
- Using `setContent` for emails (zero network)
- Using `waitUntil: 'networkidle'` for web targets
- Using `document.fonts.ready` so font metrics stabilise before measuring
- Quantising colours by `n → Math.round(n / 32) * 32`
- Using only regex / DOM / computed-style signals (no LLM, no randomness)

If a re-run drifts more than ±0.02 on any aggregate, it's a bug in the runner,
not the content — file under SCRIPT specialist.
